---
title: Architecture
description: A layered, lightly-hexagonal design on top of a Loco skeleton.
---

`latch` runs as a single [Loco](https://loco.rs) (axum) binary. Loco is MVC by default, but
`latch` doesn't follow that naively: two entry points (the admin API and the MCP tool) run the
**same** business operation ‚Äî both deploy and list. So a **service layer** sits between
controllers and models. The result is a layered, lightly-hexagonal design.

## The layers

<Cards>
  <Card title="The core ‚Äî src/services/">
    HTTP-agnostic. The only layer that holds business logic. It knows nothing about requests,
    cookies, tokens, or axum. Returns a plain `CoreError`.
  </Card>
  <Card title="Inbound adapters (thin)">
    Translate a request into a service call: web controllers, MCP tools, client serving. They
    carry every auth decision.
  </Card>
  <Card title="Outbound adapters">
    SeaORM (projects, versions, the active-version pointer) and a pluggable `Storage` trait for
    the HTML files.
  </Card>
</Cards>

<Callout type="info">
The rule that holds it together: **the core assumes its caller is already authorized.** Every
auth decision is made *before* entering a service ‚Äî session on the admin, `deploy_token` in the
MCP tool, unlock code in the client serving. A service like `deploy()` never sees a transport
secret. That's what makes the adapters interchangeable over one core, and the core testable
without HTTP.
</Callout>

## Errors

The core returns a `CoreError` (via `thiserror`), with no axum or Loco types. Each adapter maps
it: web ‚Üí HTTP status + JSON; MCP ‚Üí a tool error. Loco keeps its own `Error` for its plumbing;
the core ignores it.

## Data model

Two tables hold the domain, plus a session store:

- **`projects`** ‚Äî `id`, unique `slug` (readable base + random suffix), `name`,
  `code_enabled` (true by default), `pin` (6 digits, recoverable so the admin can copy it),
  `brand_name` (optional), `active_version_id` (nullable until the first deploy).
- **`versions`** ‚Äî `id`, `project_id`, `n` (per-project incrementing number), `html_path`
  (relative path in the storage volume), `release_notes` (optional Markdown text, max 10 000
  chars, stored raw ‚Äî never rendered server-side), `created_at`.
- **`sessions`** ‚Äî the admin session store.

## `deploy()` ‚Äî an imposed order

Deploys are ordered to never leave a project pointing at a half-written file:

<Steps>
<Step>
**Write the HTML first** ‚Äî a temporary name, then an atomic rename into place, through the
`Storage` trait.
</Step>
<Step>
**Then the transaction** ‚Äî insert the `versions` row (with optional `release_notes`) and, if
`activate`, flip `active_version_id`.
</Step>
</Steps>

If the database fails after the write, you get an orphan file (harmless, collectable). The
reverse order would point an active version at a missing file ‚Äî the worst state for a client.

The admin's manual deploy and the MCP `deploy_prototype` tool call the **same** `deploy()`.

## Client serving ‚Äî shell + iframe

`GET /c/<slug>` always serves a **shell page** that loads the prototype inside an `<iframe
src="/c/<slug>/raw">`. This two-layer structure lets the shell overlay release notes on top of
the prototype without touching the prototype's HTML.

### Endpoints under `/c/<slug>`

| Endpoint | Purpose | Auth |
| --- | --- | --- |
| `GET /c/<slug>` | Shell page (or unlock page if protected) | Unlock cookie if code-protected |
| `POST /c/<slug>/unlock` | Verify PIN, issue signed unlock cookie | Public (rate-limited) |
| `GET /c/<slug>/raw` | Prototype HTML (iframe target) | Same unlock gate; `frame-ancestors 'self'`, `no-store` |
| `GET /c/<slug>/notes` | `{ n, notes_md }` or `204` (no notes) | Same unlock gate; `no-store` |

The `/raw` and `/notes` endpoints carry the same gate as the shell: a code-protected prototype
that hasn't been unlocked returns `403` ‚Äî release notes are not leaked before authentication.

### Release notes overlay

After gaining access the shell fetches `/c/<slug>/notes`. If the response contains `notes_md`
and `localStorage['latch:seen:<slug>']` doesn't match the current version number `n`, the shell
renders the notes in a full-screen overlay using the restricted Markdown renderer (see
[Security model](/docs/how-it-works/security-model#markdown-rendering)). Dismissing the overlay
updates `localStorage` ‚Äî the overlay won't reappear for the same version.

<Callout type="warn">
All prototypes now run inside an iframe. Code in the prototype that relies on `window.top`,
fullscreen APIs, or behaviours that differ in an embedded context may need adjustment.
</Callout>
