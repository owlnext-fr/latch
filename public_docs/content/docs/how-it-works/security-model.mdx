`latch` is meant to *hide* prototypes, but the real gating is authentication: an admin session,
the MCP `deploy_token`, and an optional per-project code. Crawlers are kept out with
`robots.txt` (`Disallow: /`) and an `X-Robots-Tag: noindex, nofollow` header, but those are
hints ŌĆö the security comes from the layers below.

## Hard invariants

These are non-negotiable and covered by tests that break the build if violated:

<Callout type="warn">
1. **No response ever returns a hash** ŌĆö not on the web, not over MCP. Guaranteed structurally:
   no admin DTO carries a hash field.
2. **The cleartext PIN only appears on a project's detail view** ŌĆö never in a list, never over
   MCP. The list type simply has no `pin` field.
3. **The `deploy_token` is validated on every MCP tool**, reads included.
4. **Auth lives in the adapter, never in the core.** A service assumes its caller is authorized;
   an architecture test fails the build if `axum` or `loco_rs` is imported under `src/services/`.
5. **The `owner_token` of a comment thread is never serialized** ŌĆö responses return a boolean
   `editable` flag instead. No client (visitor or admin) ever receives the raw token.
</Callout>

## Two cookies

- **Admin session** ŌĆö a same-origin session cookie (signed with `SESSION_SECRET`), `HttpOnly`
  always, `Secure` + `__Host-` prefix in production, `SameSite=Lax`. Not the Loco users/JWT
  system ŌĆö a single account compared in constant time against `ADMIN_USER` / `ADMIN_PASS`.
  Logout destroys the session server-side.
- **Unlock cookie** ŌĆö issued after a correct PIN on `/c/<slug>/unlock`. Signed (HMAC over the
  slug + an imprint of the current PIN), `HttpOnly` + `Secure` + `SameSite=Lax`, scoped to
  `Path=/c/<slug>`. No server-side state. **Rotating the project's PIN revokes every cookie.**

## Comment identity cookie

When comments are enabled for a project, the shell issues a third cookie:

- **`latch_comment`** ŌĆö a signed identity cookie (HMAC, reuses `UNLOCK_COOKIE_SECRET`),
  `HttpOnly + Secure + SameSite=Lax`. It carries the reviewer's display name and a
  per-browser identity token. No user account is created. The cookie is issued lazily on the
  first comment the reviewer posts.

### Comment gating

Comment endpoints require **both** conditions to hold:

- The **unlock gate** ŌĆö the prototype must be accessible (no unsatisfied PIN check).
- **`comments_enabled`** on the project ŌĆö the admin has not disabled comments.

If either condition fails, comment reads and writes return `403`.

### The `owner_token` invariant

Each comment thread holds an `owner_token` used to gate edits and deletes server-side. This
token is **never sent to any client** ŌĆö not to the reviewer, not to the admin, not over MCP.
Responses expose a boolean `editable` flag derived server-side from the cookie's identity.
This is hard invariant #5 above, covered by tests that break the build if violated.

## Load-bearing rate-limits

A 6-digit PIN is 10ŌüČ combinations ŌĆö brute-forceable in seconds without a guard. So the rate-limit
on `/c/<slug>/unlock` is **load-bearing**, not cosmetic: a per-IP+slug backoff plus a global
per-slug ceiling (via `tower_governor`). The admin login is rate-limited the same way.
`SmartIpKeyExtractor` reads `X-Forwarded-For` / `X-Real-IP`, so a reverse proxy must forward
them (see [Reverse proxy](/docs/deploy/reverse-proxy)).

<Callout type="info">
The governor counters are **in-memory** and reset on reboot ŌĆö an accepted trade-off for v1.
</Callout>

## CSRF & host validation

- Every **mutating** admin endpoint checks the `Origin` header (same-origin) via a middleware
  that returns `403` cross-origin ŌĆö on top of `SameSite`.
- The MCP transport validates the `Host` header against `allowed_hosts`, derived from
  `LATCH_PUBLIC_BASE_URL`. This closes a DNS-rebinding vector (CVE-class issue in older `rmcp`);
  `rmcp` is pinned to a floor of `1.4`.

## Markdown rendering

Release notes are stored as raw Markdown text ŌĆö the server **never** renders them to HTML.
Rendering happens exclusively on the client (browser), using a **restricted** `react-markdown`
instance configured with `skipHtml` and an explicit `allowedElements` allow-list:

| Allowed | Blocked |
| --- | --- |
| Headings, paragraphs, bold, italic | Links (`<a>`) |
| Unordered and ordered lists | Images (`<img>`) |
| Blockquotes | Code blocks and inline code |
| ŌĆö | Raw HTML |

This allow-list is shared between the admin preview (deploy side-panel) and the visitor overlay,
so what the admin sees is exactly what the visitor gets.

The `/c/<slug>/notes` endpoint is gated by the same unlock check as the shell and `/c/<slug>/raw`:
a code-protected prototype that has not been unlocked returns `403` ŌĆö release notes are not
exposed before authentication.

### `frame-ancestors 'self'` on `/raw`

The prototype HTML served at `/c/<slug>/raw` carries `Content-Security-Policy: frame-ancestors
'self'`. This prevents any third-party page from embedding the prototype in its own `<iframe>`,
ensuring the shell's unlock gate cannot be bypassed by embedding `/raw` directly elsewhere.

## Fail-secure secrets

In production the boot **refuses to start** if a required secret is missing ŌĆö `ADMIN_USER`,
`ADMIN_PASS`, `DEPLOY_TOKEN`, `LATCH_PUBLIC_BASE_URL`, `SESSION_SECRET`, `UNLOCK_COOKIE_SECRET`.
There is no insecure fallback outside development. See [Configuration](/docs/deploy/configuration).
