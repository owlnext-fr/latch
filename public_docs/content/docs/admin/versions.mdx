A project is a series of **versions**. One of them is the **active** version — the one served at
`/c/<slug>`. Versions are numbered per project (`v1`, `v2`, …).

## Deploy a new version

From the detail view, **Deploy** opens a side-panel. Upload a single-file HTML prototype and
choose whether to **activate immediately**. The same operation runs whether you deploy here or
[from Claude](/docs/publish-from-claude/connect-mcp) — it's the one `deploy()` path.

<Callout type="info">
The HTML is written to storage **first**, then the version row and the active-version pointer are
updated in a transaction. A client never sees a pointer aimed at a half-written file.
</Callout>

## Release notes

Each version can carry optional **release notes** — a short Markdown text displayed to the
visitor as an overlay the first time they open the prototype after a new version is published.

### Entering notes when deploying

The deploy side-panel includes a lightweight WYSIWYG editor (Tiptap) for composing release
notes. A **Preview** tab renders the notes exactly as visitors will see them, so you can check
formatting before deploying.

**Supported Markdown:**

| Element | Syntax |
| --- | --- |
| Paragraph | plain text |
| Heading | `## Heading` |
| Bold | `**bold**` |
| Italic | `*italic*` |
| Unordered list | `- item` |
| Ordered list | `1. item` |
| Blockquote | `> note` |

**Not rendered:** links, images, code blocks, and raw HTML are intentionally ignored — the
renderer uses a restricted allow-list to block XSS vectors. Write plain descriptive text.

<Callout type="info">
Notes are capped at **10 000 characters**. Longer content is rejected with a validation error
before the deploy is saved.
</Callout>

### Version indicator

Rows in the version list show a notes icon when a version has release notes attached, so
you can tell at a glance which versions carried a message to visitors.

## How visitors see release notes

When a visitor opens `/c/<slug>` for a new version (one they haven't seen yet), the shell
displays an overlay with the release notes rendered in Markdown. Dismissing the overlay stores
the current version number in `localStorage` under the key `latch:seen:<slug>` — the overlay
will not appear again for the same version.

## Per-version actions

Each row in the version list offers:

- **Activate** — flip the active pointer to this version (a transactional update). The client link
  immediately serves it.
- **Preview** — open the raw HTML through an admin-only, `no-store` route (behind your session) —
  handy to check a version without making it active.
- **Details** — open a read-only side-panel showing the version number, deploy date, status, and
  the release notes rendered exactly as visitors see them.
- **Delete** — remove a version through a danger side-panel. Deleting the **active** version is
  refused (HTTP 400); activate another one first.

## First deploy

Before the first deploy a project has no active version, so `/c/<slug>` returns a 404 page. The
detail view surfaces the deploy panel front-and-center until you ship `v1`.
